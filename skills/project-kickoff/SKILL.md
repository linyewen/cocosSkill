# 项目启动流程

编程是艺术，游戏更是。动手之前必须先理解、再设计、再实施。跳过思考直接编码，无论多快都是浪费。

## 判断项目类型：$ARGUMENTS

---

## A. 复刻项目（从旧项目迁移）

### 第一步：理解原版（WHY，不只是 WHAT）

逐层阅读原版项目，回答以下问题：

1. **场景结构**
   - Canvas 下有哪些节点？完整列出名称、父子关系、渲染顺序
   - 每个节点的 UITransform 尺寸是多少？有没有 Widget？Widget 配置是什么？
   - 哪些节点是纯容器（逻辑节点）？哪些有视觉表现？
   - 特殊节点（非标准尺寸、特殊定位）有哪些？为什么这样设计？

2. **关键设计决策**
   - 某个节点为什么是 A 的子节点而不是 B 的？（父子关系 = 坐标系 = 游戏逻辑）
   - 哪些层有 Widget 适配？哪些故意没有？为什么？
   - 触摸事件谁处理？数据怎么流转？

3. **列出疑惑**
   - 有没有看不懂的设计？先记录，不要猜测后动手
   - 有没有觉得可以改进的地方？记录下来，复刻完成后再讨论

### 第二步：1:1 复刻

- **先场景后代码**：先把节点树 100% 还原（名称、层级、尺寸、Widget、组件），再写脚本逻辑
- **不改设计**：复刻阶段不做任何"优化"，即使觉得原版不好也先忠实还原
- **逐节点核对**：每完成一个模块，对照原版检查是否遗漏
- **API 适配**：只做引擎版本差异的 API 替换（2.x → 3.x），不改架构

### 第三步：验证 + 提出改进

复刻完成并验证功能一致后：

1. **列出原版设计的优点** — 哪些设计很精妙，值得学习
2. **列出改进建议** — 基于 3.x 的能力，哪些可以做得更好
3. **说明每个改进的理由** — 不是"我觉得"，而是"因为 3.x 提供了 XX，可以解决原版的 YY 问题"
4. **让用户决定** — 改不改、改哪些，由用户判断

## B. 新项目（从策划案/文档开始）

### 第一步：阅读全部资料

1. **策划文档**：通读游戏玩法、核心循环、数值设计
2. **美术资源**：浏览所有效果图、UI 稿、角色/场景素材，理解视觉风格
3. **参考项目**：如果有参考游戏或代码，先读懂

### 第二步：提出疑惑

在动手之前，列出所有不确定的点：

```
疑惑清单：
？ 策划案说"敌人从四面八方出现"，是指 360° 随机还是预设方向？
？ 效果图中这个按钮的交互逻辑是什么？点击后发生什么？
？ 血条是跟随敌人的还是固定在 HUD 上？
？ 关卡之间怎么过渡？有加载画面吗？
```

**不带着疑惑动手。** 问清楚再开始，比做完了返工快十倍。

### 第三步：设计方案

回答以下问题后再写代码：

1. **场景层级设计** — Canvas 下分几层？每层职责是什么？用 `/cocos-layer` 分析
2. **核心系统架构** — 用 `/feature-design` 分析每个核心功能的调用链和职责边界
3. **预制体规划** — 需要哪些预制体？用 `/prefab-analyze` 推导结构
4. **技术选型** — 物理引擎要不要开？用 2D 还是 3D 碰撞？动画用帧动画还是骨骼？

### 第四步：确认方案后动手

将方案呈现给用户，确认无误后再开始实施。

## 核心原则

- **不理解就不动手** — 看不懂的设计先问，不要猜
- **父子关系即坐标系** — 改变节点归属 = 改变游戏逻辑，这不是"整理代码"
- **特殊设计必有原因** — 某个层没有 Widget、某个节点尺寸特殊，一定是故意的，找到原因
- **复刻不是搬运** — 搬运工不理解 why，所以必然丢东西。理解了才能完整复刻
- **优化基于理解** — 没有理解就没有资格谈优化，否则叫篡改

---

## 树形思维（全局影响意识）

代码和场景都是树结构——每个节点都和父节点、子节点、兄弟节点、引用方构成一张网。**修改任何一个点，必须检查这张网上所有相关的点。**

这不是复刻项目的特殊要求，而是所有开发的基本原则：

- **改函数签名** → 检查所有调用方
- **改节点父子关系** → 检查所有 @property 绑定和坐标转换
- **改配置表字段** → 检查所有读这个字段的脚本
- **改渲染顺序** → 检查层级遮挡和点击穿透
- **改组件属性** → 检查运行时谁在读这个属性

检查清单（每次修改后执行）：
1. **向上看**：父节点的引用是否还正确？
2. **向下看**：子节点的依赖是否还满足？
3. **向左右看**：兄弟节点的顺序和关系是否受影响？
4. **向外看**：其他文件（脚本/场景/预制体）中谁引用了这个节点？

算力花在检查上不是浪费——花在返工上才是浪费。

## 从用途推导结构（最关键的思维方式）

创建任何节点时，必须先问：**这个东西在游戏里是干什么的？**

从用途出发，推导它需要什么子节点、什么组件、什么数据：

| 用途 | 推导 | 结论 |
|------|------|------|
| 贝塞尔路径 | 数学上需要起点+两个控制点+终点 | 必须有 p1/c1/c2/p2 四个子节点 |
| 手指引导轨迹 | 手势是连续的曲线，4个点不够描述 | 需要多个离散路径点依次连线 |
| 背景大地图 | 无限滚动，不能一次加载全部 | 需要动态瓦片系统，viewRadius 控制可见范围 |
| 屏幕适配层 | 不同设备屏幕比例不同 | 必须有 Widget 拉满四边 |
| 金币掉落层 | 固定在屏幕底部特定区域 | 特殊尺寸 + 固定 Y 坐标，不用 Widget |

**同名不同义的陷阱**：PathMgr 的 path 和 GuidePrefab 的 path 都叫"path"，但设计目的完全不同——前者是贝塞尔曲线（4控制点），后者是离散路径（多点连线）。不能看到名字一样就套用同一种结构。

**空节点 = 有理由**：如果原版放了一个"看起来没用"的空节点或多个占位点，先想它在运行时会被怎么用。游戏开发中没有真正多余的节点——每个节点都承载着某种运行时逻辑或数据。
