# 功能设计分析

在写任何代码之前，先完成以下分析。目的：像资深工程师一样理解设计意图，而不是机械翻译代码。

## 分析目标：$ARGUMENTS

---

### 第一步：功能链路（从触发到结束）

追踪这个功能的完整调用链，从用户操作或系统触发开始，到最终效果结束：

```
触发点 → 中间环节 → ... → 最终效果
例: 玩家点击射击按钮 → WeaponSystem.fire() → weapon.shoot() → instantiate(bulletPrefab)
    → Bullet.update() 移动 → 碰撞检测 → Enemy.takeDamage() → 死亡特效 + 掉落
```

**必须回答：**
- 入口在哪？谁触发的？
- 经过哪些类、哪些方法？
- 创建了哪些节点？挂在哪个 Layer？
- 最终效果是什么？怎么结束/销毁？

### 第二步：职责边界（谁做什么、不做什么）

列出这个功能涉及的每个类，明确它的职责边界：

```
| 类名 | 只负责 | 不负责（由谁负责） |
|------|--------|------------------|
| WeaponSystem | 管理武器集合、切换、升级 | 不负责子弹移动（Bullet自己管） |
| Bullet | 移动、碰撞、销毁自己 | 不负责伤害计算（Enemy.takeDamage管） |
| EnemyManager | 创建/回收敌人 | 不负责敌人行为（Enemy自己管） |
```

**检查点：**
- 有没有一个类做了太多事？→ 需要拆分
- 有没有两个类做了同一件事？→ 需要合并
- 创建者和使用者是否分离？→ Manager 创建，实体自己管行为

### 第三步：数据流（从配置到屏幕）

追踪数据从配置表到最终渲染的完整路径：

```
配置表 JSON → ConfigManager 解析 → TableItem 对象
  → init(cfg) 初始化实体 → 运行时状态(hp, speed, position)
    → 渲染表现(Sprite, 动画, 位置, 大小)
```

**必须回答：**
- 哪些值来自配置表？哪些是运行时计算的？
- 尺寸从哪来？（图片实际尺寸 × 配置表 scale，不是占位值）
- 速度/伤害/生命值的计算公式是什么？

### 第四步：设计模式识别

这个功能用了或应该用什么设计模式？

| 模式 | 适用场景 | 本功能是否适用 |
|------|---------|--------------|
| 策略模式 | 同一行为有多种实现（移动方式、攻击方式） | ? |
| 工厂模式 | 根据类型创建不同实例（武器、敌人） | ? |
| 观察者模式 | 事件通知（死亡、拾取、状态变化） | ? |
| 对象池 | 频繁创建销毁的实体（子弹、特效） | ? |
| 组合模式 | "有什么"比"是什么"更重要（武器系统） | ? |

### 第五步：迁移专用（如果是从旧项目迁移）

如果是迁移任务，额外回答：

1. **原项目的设计意图是什么？** — 不是"代码怎么写的"，而是"为什么这样设计"
2. **哪些是好设计要保留？** — 如 WeaponSystem + Factory 模式
3. **哪些是历史包袱要丢弃？** — 如过时的 API 用法、不合理的耦合
4. **引擎差异在哪？** — 2.x vs 3.x / Laya vs Cocos 的 API 和架构差异
5. **每个功能是否完整实现？** — 不留空桩，不写 `// TODO`，一次性完成

### 第六步：输出实施方案

分析完成后，输出：

1. **需要创建/修改的文件清单**（按依赖顺序排列）
2. **每个文件的核心逻辑**（不是详细代码，是设计决策）
3. **prefab 结构**（如需要，调用 `/prefab-analyze` 进一步推导）
4. **验证清单** — 这个功能做完后，用什么行为验证它是正确的？

```
验证清单:
☐ 敌人从屏幕上方出现并向玩家移动
☐ 敌人每隔 spantime 秒发射一颗子弹
☐ 子弹命中玩家时扣血
☐ 敌人死亡时播放爆炸特效并掉落金币
```

**确认方案无遗漏后，再开始写代码。**
